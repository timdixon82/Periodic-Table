# Decision Record 005: Release automation configuration

## Status

Accepted. Decided by Jacob (architect) on 2026-05-22, during work 003-periodic-table-accessibility. Reviewed by Sonja. Tag format (Q11A) and first-release fallback (Q12A) decided by Tim Dixon.

## Context

The Periodic-Table repository has a `Release` workflow at `.github/workflows/release.yml`. The workflow calls `googleapis/release-please-action` at version 5 (action v5), which runs release-please version 17 (release-please v17). Release-please v17 operates in manifest mode: it reads two configuration files in the repository root, `release-please-config.json` and `.release-please-manifest.json`, before it can do anything. Neither file existed.

As a result, the `Release` workflow had never been able to run. Every push to `main` since the workflow was added failed at the first step. The commit `6ab5054`, which is the Phase 2 squash merge, exposed the failure because it was the first push to `main` after the workflow was in place.

This record decides how release-please is configured to fix the failure and to record the release automation choices that every future release of this project depends on.

## Decision

### Release type: `simple`

The release type is set to `simple`. The `simple` strategy is release-please's generic strategy for a project whose version is a plain string in a file. It has no language-specific assumptions, expects no product package manifest, and runs no build step.

Periodic-Table is a static front-end site with no build step (Decision Record 002) and no product `package.json` at the repository root. A `package.json` exists only inside `node_modules/` for the development-only lint and test tooling, but the project itself ships no `package.json`. The `node` release type, which would expect a root `package.json` with a `version` field, therefore does not apply.

The project already tracks its version in a `VERSION` file at the repository root. The `simple` strategy's default version file is named `version.txt`. Rather than rename `VERSION`, the configuration uses the `extra-files` generic updater (see below) to point release-please at the existing `VERSION` file. This means the `VERSION` file is kept in sync automatically on every release, and no version file is ever edited by hand.

### The two manifest files

Two files are added to the repository root.

`release-please-config.json` is the release-please configuration file. It sets the release type to `simple`, turns off component prefixes in the tag, turns on the `v` prefix in the tag (see tag format below), sets the changelog path to `CHANGELOG.md`, and uses an `extra-files` generic entry to point at `VERSION` so that file is bumped automatically on each release.

`.release-please-manifest.json` is the manifest file. It records the version that release-please treats as the last released version. It is seeded at `1.0.0`. This value agrees with the `VERSION` file on `main`, and with the `Release-As: 1.0.0` footer on commit `6ab5054`. Setting the manifest to `1.0.0` means release-please sees "last released 1.0.0" and "this commit requests release as 1.0.0", so it opens the 1.0.0 release pull request cleanly, with no version going backwards and no conflict.

### Tag format: `v` prefix

Tags are formatted as `v1.0.0`, with a `v` prefix. This is the release-please default and the common convention for version tags. Tim decided this at Q11A.

### Changelog

The changelog is generated by release-please into `CHANGELOG.md` at the repository root. Release-please reads the Conventional Commits history since the last release and writes a new section into `CHANGELOG.md` inside each release pull request. No `CHANGELOG.md` is committed by hand before the first release; a hand-committed file would clash with the one release-please creates.

This carries forward the Phase 1 changelog decision, recorded in the Phase 1 release checklist at `.claude/work/001-periodic-table-setup/`, which states that the changelog is generated by release-please at release time and that no pre-written `CHANGELOG.md` is required.

### First-release fallback

With the manifest seeded at `1.0.0`, release-please should produce the 1.0.0 release pull request because `6ab5054` carries a `Release-As: 1.0.0` footer. If the first `Release` workflow run after this fix is silent instead, Sean is pre-authorised to add one empty commit to `main` with a `Release-As: 1.0.0` footer to force the release pull request. This action is recorded in the work log. Tim decided this at Q12A.

### Change to `release.yml`

The `with:` block of the `Release Please` step in `.github/workflows/release.yml` is updated to pass `config-file` and `manifest-file` explicitly. This tells the action exactly where to find the two new files, and means the workflow will not fail silently if the action's default file names ever change. No other part of `release.yml` is changed: the trigger, the permissions, the runner, and the SHA-pinned action reference are all preserved.

## Relationship to other records

This record is a sibling of Decision Records 001 to 004. It supersedes nothing. It records a previously unrecorded decision and gives the project a single durable home for how this project releases.

## Alternatives considered

### Release type `node`

Rejected. The `node` type expects a `package.json` at the repository root with a `version` field. Periodic-Table has no such file. Using `node` would cause release-please to fail on every run.

### Rename `VERSION` to `version.txt`

Considered but not chosen. The `simple` strategy's default version file is `version.txt`. Renaming `VERSION` would remove the need for the `extra-files` entry. It is not chosen because `VERSION` is already referenced in the global `release-process.md` and in the Phase 1 release checklist, and renaming it is needless churn. Keeping `VERSION` and pointing `extra-files` at it is the cleaner fix.

### Tag format without `v` prefix (option Q11B)

Considered. An unprefixed tag such as `1.0.0` would match the bare value in the `VERSION` file exactly. Not chosen; Tim selected the `v` prefix at Q11A.

### Seed the manifest at `0.0.0`

Considered. Starting at `0.0.0` would also work because the `Release-As: 1.0.0` footer overrides the computed bump. Not chosen because it would briefly misstate history: the manifest would claim the last release was `0.0.0` when no release has ever been made and the `VERSION` file already reads `1.0.0`. Starting at the value the project already declares is the honest choice.

## Consequences

- Release-please can now run. Every push to `main` triggers the `Release` workflow, which reads the two new configuration files and proposes the next release pull request.
- The `VERSION` file is kept in sync automatically by the `extra-files` generic entry. It is never edited by hand.
- The `CHANGELOG.md` is generated by release-please from the Conventional Commits history. It is never written by hand.
- The first release will be tagged `v1.0.0`. The 1.0.0 release pull request will include the changelog for all commits from the start of history to `6ab5054`.
- If the first `Release` workflow run is silent, Sean adds one empty commit with a `Release-As: 1.0.0` footer and records it in the work log.
